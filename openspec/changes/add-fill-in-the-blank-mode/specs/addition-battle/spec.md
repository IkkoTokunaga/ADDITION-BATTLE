## ADDED Requirements

### Requirement: ゲームモードの選択（通常／虫食い）
システムは、スタート画面において「通常モード」と「虫食いモード」の2つの開始ボタンを提供し、プレイヤーが選択したモードでバトルを開始できるようにしなければならない（MUST）。選択されたモード（`normal` または `blank`）は、暗号化セッショントークン・ステージ進行用キャリートークン・スコア保存まで一貫して引き継がれなければならない。ニックネーム入力なしのゲスト即時開始は維持しなければならない。

#### Scenario: 通常モードでの開始
- **WHEN** プレイヤーがスタート画面で「通常モード」の開始ボタンを押したとき
- **THEN** ニックネーム入力を求めることなくゲーム画面へ遷移し、和を答える通常モードでバトルを開始しなければならない。発行されるセッショントークンには `mode = normal` が含まれなければならない。

#### Scenario: 虫食いモードでの開始
- **WHEN** プレイヤーがスタート画面で「虫食いモード」の開始ボタンを押したとき
- **THEN** ニックネーム入力を求めることなくゲーム画面へ遷移し、隠された数を答える虫食いモードでバトルを開始しなければならない。発行されるセッショントークンには `mode = blank` が含まれなければならない。

#### Scenario: ステージ進行時のモード維持
- **WHEN** プレイヤーがステージをクリアして次のステージへ進行したとき
- **THEN** 開始時に選択したモードが次ステージでも維持されなければならない（キャリートークンにモードが含まれ、新ステージのトークンへ引き継がれなければならない）。

### Requirement: 虫食い算問題の生成と表示
システムは、虫食いモードにおいて、各ステージの難易度規則で生成した足し算 `a + b = c` の被加数（1項目）または加数（2項目）のいずれかを問題ごとにランダムに `□` として隠し、隠した数（＝正解）をクライアントへ送信せずに出題しなければならない（MUST）。和（`c`）自体を隠してはならない。

#### Scenario: □位置のランダムな決定
- **WHEN** 虫食いモードで1問が生成されたとき
- **THEN** その問題の `□` 位置は被加数（1項目）または加数（2項目）のいずれかにランダムに決定されなければならない。和を `□` にしてはならない。

#### Scenario: 隠した数を含まない公開ペイロード
- **WHEN** 虫食いモードの問題がフロントエンドへ配信されるとき
- **THEN** 公開ペイロードには表示に必要な値（見える側の項と和、`□` の位置）のみが含まれ、隠した数（正解）が含まれてはならない。正解はサーバーのセッショントークン内にのみ保持されなければならない。

#### Scenario: 虫食い式の表示
- **WHEN** 虫食いモードの問題が画面に表示されるとき
- **THEN** `□ + b = c`（被加数が隠れている場合）または `a + □ = c`（加数が隠れている場合）の形式で表示され、`□`（隠された箇所）が視覚的に強調されなければならない。プレイヤーはテンキー／キーボードで隠された数を入力できなければならない。

## MODIFIED Requirements

### Requirement: 計算問題の自動生成と非同期バッチ配信
システムは、各ステージの計算難易度規則に従い、Honoバックエンドで問題を5問ずつ自動生成し、解答を暗号化したセッショントークン (`session_token`) とともにフロントエンドへ提供し、フロントエンド側でバックグラウンドでの非同期先読みを行わなければならない。問題はゲームモード（`normal` / `blank`）に応じた形式で生成・配信され、いずれのモードでも隠された正解（通常モードの和、虫食いモードの隠した項）をトークン内にのみ保持し、公開ペイロードに含めてはならない。（MUST）

#### Scenario: 初期問題の生成とセッショントークンの発行
- **WHEN** プレイヤーがゲームまたは新しいステージを開始したとき
- **THEN** サーバーは対応するステージ難易度と選択されたモードで5問の問題を生成し、正解・問題データ・モードを含む暗号化された `session_token` と、表示に必要な情報のみ（答え抜き）の問題文をフロントエンドへ返さなければならない。

#### Scenario: 4問目回答時の非同期追加問題取得（0msディレイ）
- **WHEN** フロントエンド側で現在キューの4問目の回答が開始されたとき
- **THEN** フロントエンドは非同期通信で `session_token` をサーバーへ送信し、サーバーはトークン内のモードと同一モードで追加の5問と更新された `session_token` を生成して返さなければならない（ユーザーの画面遷移に遅延を伴ってはならない）。

### Requirement: 回答判定とダメージ処理・セッション検証
システムはプレイヤー（数学の女教師）が送信した回答の正誤を判定し、正解なら鬼にダメージを与え、不正解ならプレイヤーのライフ（初期値はハート3個）を1つ減少させなければならない。正解の定義はモードに従い、通常モードでは「和」を、虫食いモードでは「隠された項（1項目なら被加数、2項目なら加数）」を正解とする。チート防止のため、判定結果の最終整合性はサーバー側でセッショントークンを復号し、各問題のモードと `□` 位置に基づいて検証されなければならない。スコア・ダメージ・タイムボーナス・必殺技ゲージの計算式は両モードで共通でなければならない。（MUST）

#### Scenario: 正解による鬼へのダメージとタイムボーナス加算
- **WHEN** プレイヤーが正しい回答（通常モードは和、虫食いモードは隠された項）を入力して決定ボタン（またはエンターキー）を押したとき
- **THEN** プレイヤーのスコアが加算され、鬼のHPに同等のダメージ（基本得点：ステージ×100 ＋ タイムボーナス）が与えられ、ダメージ演出が表示されて次の問題へ進まなければならない。回答時間が10秒未満（$t < 10$）のときはタイムボーナス $(10 - t) \times ステージ / 10$ 点が自動的に加算されなければならない（タイムボーナスはプレイヤーには非表示）。

#### Scenario: 不正解によるプレイヤーへのダメージ（毒攻撃）
- **WHEN** プレイヤーが誤った回答を入力して決定ボタン（またはエンターキー）を押したとき
- **THEN** 鬼から毒攻撃を受ける演出が発生し、プレイヤーのハート型ライフが1つ減少し、入力フィールドがクリアされなければならない。

#### Scenario: ライフ消滅によるゲームオーバー
- **WHEN** プレイヤーのライフ（ハート）が0個になったとき
- **THEN** ゲームオーバー演出が表示され、新たな回答の入力が受け付けられなくならなければならない。

#### Scenario: モードと□位置に基づくサーバー再検証
- **WHEN** クリアまたはゲームオーバー時にサーバーが `session_token` と解答リストを再検証するとき
- **THEN** サーバーは各問題を `(被加数, 加数, □位置)` の組として照合し、トークンが生成していない組を参照する解答を改ざんとして検証失敗にしなければならない。正誤は各問題のモード・`□` 位置に基づいて算出され、スコア計算は通常モードと同一の式で行われなければならない。

### Requirement: スコアおよび進行状況・出題履歴の保存
システムは、ゲームオーバー時（またはステージクリア時）に、プレイヤーのスコア情報および実際に出題された問題・解答履歴（数、回答値、正誤、所要時間）を Neon PostgreSQL データベースへ Hono バックエンド API 経由で保存しなければならない（MUST）。スコアレコードにはプレイしたゲームモード（`normal` / `blank`）を記録しなければならない。保存するモードはクライアント申告ではなくセッショントークン内のモードを真実としなければならない。出題履歴の書き込みは非同期で行い、ゲーム進行をブロックしてはならない。

#### Scenario: ゲームオーバー時のデータ保存と改ざん検証・履歴ログの記録
- **WHEN** プレイヤーのライフが0になりゲームオーバーになり、スコア保存リクエストが送られたとき
- **THEN** サーバーは `session_token` を復号してこれまでの全解答と回答時間の正当性をモードと `□` 位置に基づいて検証し、検証に成功した場合のみ、スコアレコード（`player_id`、`username`、最終スコア、到達ステージ、`mode`）をデータベースに保存し、同時に関連するすべての出題履歴（`num1`、`num2`、`user_answer`、`is_correct`、`time_taken`）を非同期（レスポンスを待たせない形）で `question_logs` テーブルに記録しなければならない。

#### Scenario: モード未記録レコードの後方互換
- **WHEN** モード情報を持たない既存のスコアレコードが読み出されたとき
- **THEN** システムは当該レコードを通常モード（`normal`）として扱わなければならない。

### Requirement: スタート画面のゲスト即時開始
システムは、スタート画面からニックネーム入力欄を撤去し、モードを選ぶ2つのボタン（「通常モード」「虫食いモード」）の操作のみでゲストとして即時にバトルを開始できるようにしなければならない（MUST）。ログイン・アカウント・パスワードのUIは設けない。

#### Scenario: ニックネーム入力なしでのプレイ開始
- **WHEN** スタート画面でプレイヤーがいずれかのモードの開始ボタンを押したとき
- **THEN** ニックネーム入力を求めることなくゲーム画面へ遷移し、選択したモードでバトルを開始できなければならない。表示名はゲーム終了時に確定する。

### Requirement: ランキング画面の表示
システムは、ランキングをゲームモード（`normal` / `blank`）別に分離して表示しなければならない（MUST）。ランキング画面ではモードを切り替えて閲覧でき、選択中のモードのトップ10のスコアをスクロールなしで全件表示しなければならない。加えて、ゲームオーバーまたはALLクリア後にランキング登録を行ってランキング画面へ遷移した場合には、当該プレイのモードのランキング上で今回のプレイの順位と、その前後3レコード（最大±3件）を表示し、今回のプレイのレコードを強調表示しなければならない。このとき画面の並びは、上から「今回のプレイ（順位＋前後レコード）」「スタート画面に戻るボタン」「TOP10ランキング」の順でなければならない。

#### Scenario: モード別トップ10の全件表示
- **WHEN** ランキング画面で特定のモードが選択されて表示されたとき
- **THEN** そのモードのスコアの高い順にトップ10がスクロールなしで全件表示されなければならない。他モードのスコアが混在してはならない。

#### Scenario: 登録直後の自分の順位と前後表示
- **WHEN** プレイヤーがゲームオーバーまたはALLクリア後にランキング登録を行い、その結果を伴ってランキング画面へ遷移したとき（登録済み結果の再閲覧を含む）
- **THEN** 当該プレイのモードのランキング上で、画面上部に今回のプレイの順位と、その前後3レコード（範囲端では件数が減ってよい）が順位順に表示され、今回のプレイのレコードが視覚的に強調表示されなければならない。続いて「スタート画面に戻る」ボタン、最後にそのモードのTOP10ランキングの順で表示されなければならない。

#### Scenario: 通常導線からの遷移
- **WHEN** プレイヤーがHOME画面やヘッダー等の通常導線からランキング画面へ遷移したとき
- **THEN** 自分の順位・前後表示は行わず、選択中のモード（既定は `normal`）のTOP10ランキングを表示しなければならない。
