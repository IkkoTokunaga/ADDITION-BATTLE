## Context

既存ゲームは Hono バックエンドで足し算問題を生成し、解答と問題を暗号化した `session_token`（ステートレス）に格納してフロントへ配信、クリア／ゲームオーバー時にサーバーがトークンを復号して解答・時間の整合性を再検証する設計です（チート防止）。本変更は、この設計を壊さずに「虫食い算モード」を追加します。

虫食いモードは、出題する足し算 `a + b = c` 自体は通常モードと同じ難易度規則で生成し、「画面に見せる要素」と「正解として要求する値」だけをモードで切り替えます。

## Goals / Non-Goals

**Goals:**
- 通常モード（和を答える）と虫食いモード（隠された項を答える）の2モードを提供する。
- 虫食いモードでは `□` の位置（1項目＝被加数 / 2項目＝加数）を問題ごとにランダムにする。
- 隠された数（正解）をクライアントに送らず、サーバー再検証でチートを防ぐ既存の安全性を維持する。
- スコア／ダメージ／必殺技ゲージの計算式を両モード共通に保ち、ランキングのみモードで分離する。

**Non-Goals:**
- 引き算・掛け算・割り算など他の演算の追加（対象外）。
- 和（`c`）自体を `□` にする出題（それは通常モードと等価のため作らない）。
- ステージ難易度規則そのものの変更（既存の1〜12の規則を流用）。

## Decisions

### モードの表現と引き継ぎ
- モードは `mode: 'normal' | 'blank'` の文字列で表現する。
- `PlayToken` に `mode` を追加する。`CarryToken` にも `mode` を追加し、ステージ進行（クリア時の次ステージ開始）でモードを維持する。
- `POST /api/stages/start` は `body.mode`（または `carry_token` 内の `mode`）を受け取り、未指定・不正値は `normal` にフォールバックする。

### 問題データの形（トークン内 / 公開ペイロード）
- トークン内 `Question` を拡張する:
  - `{ num1, num2, blank }`
  - `blank` ∈ `{0, 1, 2}`: `0`=和を答える（通常）、`1`=被加数（num1）が `□`、`2`=加数（num2）が `□`。
  - 通常モードは全問 `blank = 0`。虫食いモードは各問 `blank` を `1` か `2` からランダム。
- 公開ペイロード `publicQuestions` をモード／`blank` に応じて出し分け、**隠す数は絶対に含めない**:
  - `blank = 0`（通常）: `{ num1, num2, blank: 0 }` → 表示 `num1 + num2 = ?`、正解 = `num1 + num2`
  - `blank = 1`: `{ num2, sum, blank: 1 }` → 表示 `□ + num2 = sum`、正解 = `num1`
  - `blank = 2`: `{ num1, sum, blank: 2 }` → 表示 `num1 + □ = sum`、正解 = `num2`
  - ここで `sum = num1 + num2`。虫食い問題では隠す側の数値（正解）はペイロードに入れない。

### 回答判定（クライアント）
- クライアントは公開ペイロードから隠された数を逆算できる（`blank=2` なら `sum - num1`、`blank=1` なら `sum - num2`）。これは演出（命中＝ダメージ／被弾）を即時に駆動するためのローカル判定であり、最終的な得点はサーバー再検証が真実とする（通常モードでクライアントが `num1+num2` を知っているのと同じ安全モデル）。
- `answersLog` の各要素は両モードで `{ num1, num2, blank, user_answer, is_correct, time_taken }` を積む。虫食いでも `num1`/`num2` はクライアントが復元（`sum` と見えている項から算出）して送る。
- 正誤判定: `blank===0 ? user===num1+num2 : blank===1 ? user===num1 : user===num2`。

### サーバー再検証（`verifyAndScore`）
- 検証用プールのキーを `${num1}_${num2}_${blank}` に拡張する（同一 `(num1,num2)` でも `□` 位置が異なれば別問題として扱う）。
- 各提出解答について、プールに該当キーが残っていれば消費、無ければ改ざんとして `verified=false`。
- 正誤はトークン由来の `blank` で算出（クライアント申告の `blank` はプール照合で担保される）。
- スコア／タイムボーナス／鬼HP／撃破ボーナス／必殺技ゲージ加算式は変更しない。ゲージ加算は両モードとも `10 + ((num1+num2)%16)` のまま（和ベースでモード非依存）。

### スコア保存とランキング（モード別）
- `scores` テーブルに `mode VARCHAR(16) NOT NULL DEFAULT 'normal'` を追加（既存デプロイ向けに `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` のマイグレーションを `init.sql` に追記）。
- `POST /api/scores/submit` は `session_token` 内の `mode` を保存値の真実とし、`scores.mode` に記録する（クライアント申告は信用しない）。`result_hash` による二重登録防止は既存どおり（モードはトークンに含まれるためハッシュも自然に区別される）。
- `GET /api/scores` は `?mode=normal|blank`（未指定は `normal`）でフィルタしたランキングを返す。ランキング用インデックスは `(mode, score DESC, stage DESC)` を検討。
- ランキング画面はモード別に表示する（タブ切替、または URL クエリでのモード指定）。結果遷移（`?r=<score_id>`）時は当該スコアのモードで表示する。

### スタート画面（2ボタン）
- 既存の単一「修行を開始する！」ボタンを、「通常モードで開始」「虫食いモードで開始」の2ボタンに置き換える。`startNewGame(stage, mode)` のように開始時にモードを渡す。
- ニックネーム入力なしのゲスト即時開始は維持する。

## Risks / Trade-offs

- **[Risk] 隠す数の漏洩:** 公開ペイロードに隠す数を入れると虫食いの意味が無くなる。→ `publicQuestions` を `blank` で厳密に出し分け、隠す側は送らない。クライアントが `sum` から逆算できる点は通常モードと同じ安全モデル（最終得点はサーバー再検証が真実）。
- **[Risk] 検証キーの不整合:** プールキーに `blank` を含め忘れると、`□` 位置の異なる同一 `(num1,num2)` を取り違える。→ キーを `${num1}_${num2}_${blank}` に統一し、クライアント/サーバー双方で `blank` を必ず送る・検証する。
- **[Trade-off] ランキング分離の DB 変更:** `mode` 列追加が必要だが、`DEFAULT 'normal'` と後方互換マイグレーションで既存データ・既存通常モードへの影響を回避する。
- **[Risk] 後方互換:** モード未指定の古いクライアント/トークンは `normal` として扱う。`verifyAndScore` は `blank` 欠落時 `0` とみなす。
