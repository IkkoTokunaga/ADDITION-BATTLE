## Context

本プロジェクトは、足し算を解くことで鬼にダメージを与えて倒していく、ゲーミフィケーションを取り入れた学習ゲームです。教育的でありながら、インタラクティブでエンゲージメントの高い体験をユーザーに提供することを目指します。

## Goals / Non-Goals

**Goals:**
- レスポンシブで視覚的に洗練された（プレミアムなダークモード調の）ゲームUIの構築
- 計算問題の出題、テンキーおよびキーボード入力、回答判定、ダメージアニメーション、ステージ進行といったゲームコアサイクルの実装
- 主人公のライフ（ハート3個）と鬼のHP、現在のステージ、スコアといったゲーム状態のリアルタイムな更新管理
- Hono バックエンド API を介した Neon (PostgreSQL) へのスコア・ステージ情報の保存機能
- ランキング表示機能（スコアボード）

**Non-Goals:**
- 複雑な認証機能（OAuthやセッション管理など）。今回はシンプルなユーザー名入力のみでスコアを紐付けます。
- リアルタイムマルチプレイヤー対戦機能（シングルプレイヤー専用のPvEゲームとします）。
- 引き算、掛け算、割り算などの他の計算規則（ユーザー要件に基づき、今回は「足し算のみ」に限定します）。

## Decisions

- **フロントエンド: Astro + Alpine.js**
  - **理由:** Astro は静的なHTML出力が高速であり、ページ読み込みパフォーマンスに優れています。ゲームプレイ中の動的な状態管理（HPの増減、テンキーの入力、アニメーション制御）には、軽量で宣言的に記述できる Alpine.js が最適です。Reactなどの重厚なフレームワークを導入するよりもオーバーヘッドが少なく、高速に動作します。
- **バックエンド: Hono (Vercel Serverless)**
  - **理由:** Hono は非常に高速かつ軽量なWebフレームワークであり、Vercel Serverless / Cloudflare Workers 等のエッジ・サーバーレス環境と親和性が高いため、デプロイや保守が容易です。
- **データベース: Neon (PostgreSQL)**
  - **理由:** クラウド型のサーバーレス PostgreSQL であり、Vercel Serverless との相性に優れます。FaaSでのコネクションプール枯渇を防ぐため、`@neondatabase/serverless` ドライバー（WebSocket経由接続）または SQL over HTTP API を用いてステートレスにクエリを実行します。
  - **スキーマ案 (scores および question_logs テーブル):**
    - **scores テーブル (スコア記録):**
      - `id` (SERIAL, Primary Key)
      - `player_id` (VARCHAR(255), 必須。ブラウザのlocalStorageに保存されたUUID)
      - `username` (VARCHAR(50), 必須)
      - `score` (INTEGER, 必須)
      - `stage` (INTEGER, 必須)
      - `created_at` (TIMESTAMP, デフォルト現在時刻)
    - **question_logs テーブル (出題問題ログ):**
      - `id` (SERIAL, Primary Key)
      - `score_id` (INTEGER, 必須。scores.idへの外部キー)
      - `num1` (INTEGER, 必須。足される数)
      - `num2` (INTEGER, 必須。足す数)
      - `user_answer` (INTEGER, 必須。ユーザーの回答)
      - `is_correct` (BOOLEAN, 必須。正誤フラグ)
      - `time_taken` (REAL, 必須。回答にかかった時間(秒))
      - `created_at` (TIMESTAMP, デフォルト現在時刻)
  - **非同期ログ保存設計:**
    - ステージクリア時またはゲームオーバー時に、Honoサーバーが受信した解答データを検証した後、スコア保存の処理と並行して（非同期 Promise を利用してレスポンス送信をブロックせずに）`question_logs` へバルクインサート（一括挿入）を行います。これにより、ゲームのテンポを一切損なわずに出題履歴を記録します。
- **非同期問題生成と暗号化セッション検証 (チート防止)**
  - **仕様:**
    1. フロントエンドはステージ開始時に `POST /api/stages/start` を呼び出し、最初の5問（Q1〜Q5）と、その問題・解答データを暗号化した `session_token` を受け取る。
    2. ユーザーが4問目（Q4）を解き始めた時点で、フロントはバックグラウンド（非同期）で `POST /api/stages/more` を呼び出して現在トークンを送信。サーバーは追加の5問（Q6〜Q10）とそれらを合体させた新トークンを生成し、フロントに返す。この並行処理によりユーザー側のロード遅延は「0ms」となる。
    3. ステージクリア時に `POST /api/stages/clear` で解答リストと最終トークンを送信。サーバー側で解答と回答時間 $t$ 秒の整合性を一括検証する。解かれずに余った問題（例：Q11〜Q15が生成されたがQ10で撃破した場合のQ11〜15など）は、検証時に単純に無視・破棄される。
  - **メリット:** サーバー上のDBやセッションに一時的な問題データを一切書き込まない「完全ステートレス設計」のため、極めて高速かつセキュリティ強度が高く、DB接続数も最小限に抑えられます。

- **ゲーム難易度（出題ロジック）とスコア設計**
  - **ステージ出題制限:**
    - ステージ1: 1桁 + 1桁 (繰り上がりなし、和が9以下)
    - ステージ2: 1桁 + 1桁 (制限なし)
    - ステージ3: 1桁 + 2桁 (繰り上がりなし)
    - ステージ4: 1桁 + 2桁 (制限なし)
    - ステージ5: 2桁 + 2桁 (繰り上がりなし)
    - ステージ6: 2桁 + 2桁 (制限なし)
    - ステージ7: 3桁 + 2桁 (繰り上がりなし)
    - ステージ8: 3桁 + 2桁 (制限なし)
    - ステージ9: 3桁 + 3桁 (繰り上がりなし, 和が1000以下)
    - ステージ10: 3桁 + 3桁 (和が1000以下)
    - ステージ11: 3桁 + 3桁 (制限なし)
    - ステージ12: 1〜999 + 1〜999 (制限なし)
  - **スコア/ダメージ/HP計算:**
    - **基本得点:** $ステージ数 \times 100$
    - **タイムボーナス:** $t < 10$秒で正解時に加算。計算式：$(10 - t) \times ステージ数 / 10$ （※回答者には非表示、基本得点と合算して表示）。時間切れによるペナルティ（ライフ減少等）は設けず、時間は得点ボーナスにのみ影響する。
    - **鬼のHP:** $ステージ数 \times 700$ (基本得点ベースで概ね7問前後の正解で撃破できる、達成感とテンポを両立した難易度設定)
    - **鬼ダメージ量:** プレイヤーが正解して得たスコア値そのもの
    - **鬼撃破ボーナス:** 鬼の最大HP / 2。撃破時にUI上で追加加算される。

- **音響効果 (BGM & 効果音) 設計**
  - **BGMとView Transitions:** Astro標準の `<ViewTransitions />`（または最新の `ClientRouter`）を使用し、スタート画面からゲーム画面、ランキング画面への遷移時にブラウザのフルリロードを防止します。これにより、BGM用の `<audio>` 要素を永続化（`data-astro-transition-persist`属性を指定）し、ページをまたいでもBGMを滑らかに流し続けます。
  - **Web Audio APIによる効果音（SE）の低遅延再生:** 音声アセット（`public/sounds/` 内に配置、軽量なmp3形式）をゲーム起動時に `fetch` し、Web Audio API の `AudioContext` を使ってメモリ上にデコード（キャッシュ）しておきます。これにより、回答入力時やダメージ発生時などのインタラクションに対して0msの低遅延で即座に効果音を再生します。
  - **自動再生ポリシー対策 (ユーザーインタラクション ofトリガー):** スタート画面に「ゲーム開始」ボタンを配置し、ユーザーがクリック等の明示的なアクションを行ったタイミングで `AudioContext` の再開（`resume`）およびBGMの再生開始を行います。これによりブラウザの自動再生ブロックを確実に回避します。

- **開発環境: Docker を利用したコンテナ化**
  - **理由:** ホストOS環境を汚さず開発を行い、開発者間でのバージョン（Node.jsなど）不一致を防ぐため、すべてのビルド・開発・実行コマンドは Docker コンテナ内で実行します。
  - **構成:** `Dockerfile` と `docker-compose.yml` をプロジェクトのルートに配置し、ソースコードをコンテナ内にボリュームマウント（Bind Mount）してホットリロードに対応させます。

- **UIデザインと演出:**
  - **方針:** ユーザーを惹きつけるため、ダークテーマを採用し、ネオンパープル（鬼の魔力）やネオングリーン（回復・正解）などの鮮やかなアクセントカラーを使用します。
  - **キャラクターとUI:** 主人公である「数学の女教師」のイラスト（`images/女教師.png` アセット）を表示し、彼女のステータスとしてハート型（❤️）のライフインジケータ（初期値3個）を画面上部に目立たせて配置します。
  - **鬼のアセット表示:** 各ステージに対応した画像（`images/ステージ1赤鬼.png` 〜 `images/ステージ12上級鬼.png`）を画面に表示し、ステージ進行ごとに切り替えます。
  - **エフェクト演出:**
    - **攻撃（正解時）:** 鬼への攻撃時には画面や鬼の画像が揺れる（CSS Animation / shake）エフェクトや、ダメージ数値のポップアップ（floating text）演出を施し、叩いている感覚（ゲーム性）を演出します。
    - **被ダメージ（不正解時）:** 鬼から緑色や紫色の毒が飛び散るような「毒攻撃エフェクト」を表示し、ハートが1個割れて消えるアニメーションとともに画面全体に毒の霧がかかるような視覚効果を入れます。

## Risks / Trade-offs

- **[Risk] クライアント側での値の書き換えやAPIの偽装リクエスト**
  - **Mitigation (対策):** 問題生成・回答判定の正当性および時間的整合性は、暗号化セッショントークン (`session_token`) を用いてサーバー側で厳格に検証されます。そのため、クライアント側だけでハイスコアを改ざんして保存することは不可能です。
- **[Risk] サーバーレス環境でのデータベース接続のコールドスタート**
  - **Mitigation (対策):** API 呼び出しの頻度がゲームオーバー時やステージクリア時に限定されるため、多少のコールドスタート遅延は許容範囲内とします。Neon のコネクションプール機能や適切なクエリ最適化を用います。
