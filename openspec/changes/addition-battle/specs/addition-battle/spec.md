## ADDED Requirements

### Requirement: 計算問題の自動生成と非同期バッチ配信
システムは、各ステージの計算難易度規則に従い、Honoバックエンドで問題を5問ずつ自動生成し、解答を暗号化したセッショントークン (`session_token`) とともにフロントエンドへ提供し、フロントエンド側でバックグラウンドでの非同期先読みを行わなければならない。

#### Scenario: 初期問題の生成とセッショントークンの発行
- **WHEN** プレイヤーがゲームまたは新しいステージを開始したとき
- **THEN** サーバーは対応するステージ難易度で5問の問題を生成し、正解と問題データを含む暗号化された `session_token` と問題文（答え抜き）をフロントエンドへ返さなければならない。

#### Scenario: 4問目回答時の非同期追加問題取得（0msディレイ）
- **WHEN** フロントエンド側で現在キューの4問目の回答が開始されたとき
- **THEN** フロントエンドは非同期通信で `session_token` をサーバーへ送信し、サーバーは追加の5問と更新された `session_token` を生成して返さなければならない（ユーザーの画面遷移に遅延を伴ってはならない）。

### Requirement: ステージごとの計算難易度設計
システムは、現在のステージ（1〜12）に応じて、以下の規則に厳格に従って計算問題を生成しなければならない。

#### Scenario: ステージごとの数値生成規則
- **WHEN** 各ステージで問題が生成されるとき
- **THEN** 数値の範囲と繰り上がりの有無は以下の通りに決定されなければならない。
  - ステージ1: 1桁 + 1桁 (繰り上がりなし、和が9以下)
  - ステージ2: 1桁 + 1桁 (制限なし、繰り上がり可能)
  - ステージ3: 1桁 + 2桁 (繰り上がりなし)
  - ステージ4: 1桁 + 2桁 (制限なし)
  - ステージ5: 2桁 + 2桁 (繰り上がりなし)
  - ステージ6: 2桁 + 2桁 (制限なし)
  - ステージ7: 3桁 + 2桁 (繰り上がりなし)
  - ステージ8: 3桁 + 2桁 (制限なし)
  - ステージ9: 3桁 + 3桁 (繰り上がりなし, 和が1000以下)
  - ステージ10: 3桁 + 3桁 (和が1000以下)
  - ステージ11: 3桁 + 3桁 (制限なし)
  - ステージ12: 1〜999 + 1〜999 (制限なし)

### Requirement: テンキーおよびキーボードによる回答入力
システムは、画面上に0〜9の数字ボタン、バックスペース（消去）ボタン、決定（送信）ボタンで構成されるバーチャルテンキーを提供し、物理キーボードからの数字入力およびエンターキー、バックスペースキーによる入力も受け付けなければならない。

#### Scenario: 数字ボタンによる値の入力
- **WHEN** プレイヤーが画面上の数字ボタンをクリックするか、キーボードの数字キーを押したとき
- **THEN** 入力された数字が現在の回答入力フィールドの末尾に追加表示されなければならない。

#### Scenario: 1文字消去操作
- **WHEN** プレイヤーが画面上の消去ボタンをクリックするか、キーボードのバックスペースキーを押したとき
- **THEN** 現在の回答入力フィールドの最後の1文字が削除されなければならない。

### Requirement: 回答判定とダメージ処理・セッション検証
システムはプレイヤー（数学の女教師）が送信した回答の正誤を判定し、正解なら鬼にダメージを与え、不正解ならプレイヤーのライフ（初期値はハート3個）を1つ減少させなければならない。チート防止のため、判定結果の最終整合性はサーバー側でセッショントークンを復号して検証されなければならない。

#### Scenario: 正解による鬼へのダメージとタイムボーナス加算
- **WHEN** プレイヤーが正しい回答を入力して決定ボタン（またはエンターキー）を押したとき
- **THEN** プレイヤーのスコアが加算され、鬼のHPに同等のダメージ（基本得点：ステージ×100 ＋ タイムボーナス）が与えられ、ダメージ演出が表示されて次の問題へ進まなければならない。回答時間が10秒未満（$t < 10$）のときはタイムボーナス $(10 - t) \times ステージ / 10$ 点が自動的に加算されなければならない（タイムボーナスはプレイヤーには非表示）。

#### Scenario: 不正解によるプレイヤーへのダメージ（毒攻撃）
- **WHEN** プレイヤーが誤った回答を入力して決定ボタン（またはエンターキー）を押したとき
- **THEN** 鬼から毒攻撃を受ける演出が発生し、プレイヤーのハート型ライフが1つ減少し、入力フィールドがクリアされなければならない。

#### Scenario: ライフ消滅によるゲームオーバー
- **WHEN** プレイヤーのライフ（ハート）が0個になったとき
- **THEN** ゲームオーバー演出が表示され、新たな回答の入力が受け付けられなくならなければならない。

### Requirement: ステージ進行と鬼撃破ボーナス
システムは、鬼のHPが0になったときにステージクリアとし、次のステージへ進行させ、プレイヤーのライフを全回復し、撃破した鬼の最大HPの半分をボーナス得点として加算しなければならない。

#### Scenario: 鬼の撃破とステージクリア
- **WHEN** 鬼のHPが0に達したとき（鬼のHPはステージ×700に設定）
- **THEN** ステージクリアの演出（爆発演出に続くクリアモーダル）が表示され、プレイヤーの得点に撃破ボーナス（鬼の最大HP / 2）がUI上で追加加算され、ステージ数が1加算され、ライフ（ハート3個）が全回復し、新たな鬼（最大HP＝新ステージ×700）が出現しなければならない。また、前ステージで未解答のまま余った生成問題は安全に破棄されなければならない。

### Requirement: スコアおよび進行状況・出題履歴の保存
システムは、ゲームオーバー時（またはステージクリア時）に、プレイヤーのスコア情報および実際に出題された問題・解答履歴（数、回答値、正誤、所要時間）を Neon PostgreSQL データベースへ Hono バックエンド API 経由で保存しなければならない。この際、出題履歴の書き込みは非同期で行い、ゲーム進行をブロックしてはならない。

#### Scenario: ゲームオーバー時のデータ保存と改ざん検証・履歴ログの記録
- **WHEN** プレイヤーのライフが0になりゲームオーバーになり、スコア保存リクエストが送られたとき
- **THEN** サーバーは `session_token` を復号してこれまでの全解答と回答時間の正当性を検証し、検証に成功した場合のみ、スコアレコード（`player_id`、`username`、最終スコア、到達ステージ）をデータベースに保存し、同時に関連するすべての出題履歴（`num1`、`num2`、`user_answer`、`is_correct`、`time_taken`）を非同期（レスポンスを待たせない形）で `question_logs` テーブルに記録しなければならない。

### Requirement: 必殺技ゲージとカットイン演出
システムは、プレイヤーの正解回数に応じて必殺技ゲージを蓄積し、ゲージが100%に達したとき自動的に必殺技を発動して通常の3倍ダメージを与え、スラッシュ風のカットイン演出を表示しなければならない。

#### Scenario: 正解による必殺技ゲージの蓄積
- **WHEN** プレイヤーが正しい回答を入力して決定ボタン（またはエンターキー）を押したとき
- **THEN** 必殺技ゲージが一定量増加し、ゲージのUIに反映されなければならない。ゲージの加算量はサーバー側のスコア計算と同期できるよう決定論的（入力に対して固定値）でなければならない。

#### Scenario: 必殺技の自動発動とカットイン演出
- **WHEN** 必殺技ゲージが100%に達したとき
- **THEN** スラッシュ風のカットイン演出が再生され、カットイン完了後に鬼への攻撃演出が続き、通常の3倍のダメージ（スコアは通常通り）が鬼のHPに与えられなければならない。発動後、ゲージは0にリセットされなければならない。

### Requirement: バトル画面レイアウト
システムは、バトル画面において女教師と鬼のキャラクターを画面上部の左右に配置し、スコアを中央に表示するレイアウトを採用し、デスクトップ（横長）とスマートフォン（縦長）の両解像度に対応しなければならない。

#### Scenario: デスクトップレイアウト
- **WHEN** 横長の画面（デスクトップ等）でゲームが表示されたとき
- **THEN** 左上に女教師ブロック、右上に鬼ブロック（女教師より大きく表示）、中央にスコアが配置され、画面下部左に問題表示エリア・右にテンキーが並ぶレイアウトで表示されなければならない。

#### Scenario: スマートフォンレイアウト
- **WHEN** 縦長の画面（スマートフォン等）でゲームが表示されたとき
- **THEN** 上部に女教師と鬼のブロック、中央に問題表示エリア、下部にテンキーが縦に並ぶレイアウトで表示されなければならない。

### Requirement: BGMと効果音の制御と遅延のない再生
システムは、AstroのView Transitionsを活用してページ遷移時にも途切れないBGM再生を行い、Web Audio APIを使用して事前フェッチした効果音（SE）を遅延なく再生しなければならない。

#### Scenario: ユーザー操作をトリガーにしたBGM再生の開始と維持
- **WHEN** ユーザーがスタート画面の開始ボタン（クリック等の操作）をクリックしたとき
- **THEN** BGMの再生が開始され、ゲームページやランキングページへ遷移する際も、AstroのView TransitionsによりBGMが途切れることなくシームレスに再生され続けなければならない。

#### Scenario: Web Audio APIを用いた低遅延な効果音の再生
- **WHEN** 回答入力、正解（鬼への攻撃）、不正解（毒攻撃）、ステージクリア、ゲームオーバーなどのプレイヤーアクションまたは状態変化が発生したとき
- **THEN** 事前にメモリ上にデコード（キャッシュ）された対応する効果音（SE）が、ネットワークの遅延を伴うことなく即座に再生されなければならない。
